### *BCC* и “Hello World”

Теперь разберёмся, из чего состоит сама программа `hello.py`, которую мы запускаем. Как уже было сказано выше, она написана с использованием Python-библиотеки _BCC_:
```bash
#!/usr/bin/python
from bcc import BPF

program = r"""
int hello(void *ctx) {
    bpf_trace_printk("Hello World!");
    return 0;
}
"""

b = BPF(text=program)

syscall = b.get_syscall_fnname("execve")
b.attach_kprobe(event=syscall, fn_name="hello")

b.trace_print()
```

Этот код состоит из двух частей:
-  eBPF-программы, которая будет выполняться внутри ядра Linux;
-  обычного кода написанного на Python, который работает в *user space*, загружает eBPF-программу в ядро и читает вывод, который она генерирует во время выполнения.

На схеме ниже показано, что вся программа `hello.py` работает на уровне _user space_, а подпрограмма `hello()` внутри этой программы представляет собой eBPF-программу, которая будет загружена и выполнена внутри _kernel space_.


![[Pasted image 20260520133409.png#center | 430]]
На схеме показано, что обычная программа `hello.py` написанная на Python работает в *user space* и через системные вызовы загружает eBPF-программу `hello()` внутрь ядра Linux. После этого eBPF-программа начинает работать в *kernel space* и подключается к событию `execve` — системному вызову запуска новых процессов. 

Таким образом каждый раз, когда в системе запускается новая программа, ядро автоматически вызывает нашу eBPF-программу `hello()`.

Разберем каждую строку исходного кода. 

Первая строка называется _shebang_. Она сообщает операционной системе, что данный файл является Python-скриптом и должен запускаться через интерпретатор Python:
```bash
#!/usr/bin/python
```

Далее находится сама eBPF-программа. Как можно заметить, она написана на языке C:
```bash
int hello(void *ctx) {
    bpf_trace_printk("Hello World!");
    return 0;
}
```

eBPF-программа `hello()` внутри использует специальную *helper-функцию* `bpf_trace_printk()`. С её помощью программа выводит сообщение `Hello World!`.

Такие функции называются *helper functions* — вспомогательные функции ядра Linux, которые специально предоставлены для eBPF-программ. Напрямую выполнять произвольный код внутри ядра Linux нельзя, поэтому перед запуском eBPF-программы специальный механизм проверки безопасности — *verifier* _(верификатор)_ — анализирует программу и проверяет, безопасно ли её выполнять.

*Verifier* следит за тем, чтобы программа:  
-  не обращалась к случайной памяти;  
- не уходила в бесконечный цикл;  
-  не могла повредить ядро Linux.

*Helper-функции* считаются заранее безопасными, поэтому *verifier* разрешает их использование внутри eBPF-программ. В нашем случае функция `bpf_trace_printk()` выводит сообщение в специальный *tracing-буфер* ядра.

Сама eBPF-программа `hello()` подключается к событию `execve`. Это означает, что каждый раз, когда какой-либо процесс в системе вызывает системный вызов `execve()` для запуска новой программы, ядро автоматически запускает нашу eBPF-программу, и она выводит сообщение `Hello World!`.

Ключевая строка, которая сообщает библиотеке _BCC_, что следующий фрагмент кода необходимо скомпилировать и загрузить в ядро Linux, выглядит так:
```bash
b = BPF(text=program)
```

Здесь создаётся объект `BPF`, а в параметр `text` передаётся содержимое переменной `program`, внутри которой находится исходный код eBPF-программы на языке C. После этого библиотека *BCC* компилирует этот код в *eBPF-bytecode*, вызывает системный вызов `bpf()` и загружает программу в ядро Linux. Сам объект сохраняется в переменную `b`, через которую далее происходит взаимодействие с eBPF-подсистемой Linux.

![[Pasted image 20260521181254.png#center | 840]]

Я уже говорил, что eBPF-программу нужно обязательно прикрепить к определённому событию в ядре Linux. Можно представить, что мы как будто даём ей метку: #следи_за_запуском_программ. Именно поэтому eBPF-программа не запускается сама по себе, а срабатывает только тогда, когда происходит нужное событие.

В нашем примере мы прикрепляем eBPF-программу к событию `execve`. `execve()` — это системный вызов, через который в Linux запускается новая программа. Например, когда мы вводим команду `ls`, сама по себе она не появляется из воздуха: её запускает *shell*, например `bash`. Для этого `bash` вызывает системный вызов `execve()` и просит ядро Linux загрузить и запустить исполняемый файл программы `ls`. В этот момент и срабатывает наша eBPF-программа.

Сама по себе программа `strace` очень удобна и полезна, однако она работает на уровне _user space_. Из-за этого при серьёзной нагрузке и в высоконагруженных системах её используют ограниченно, поскольку она создаёт заметный *overhead* и не даёт такого уровня производительности и гибкости, как eBPF-программы.

![[Pasted image 20260521185535.png#center | 900]]

Но в нашей eBPF-программе мы точно специально указывапем называние для этого системного вызова `execve`:
```bash
syscall = b.get_syscall_fnname("execve")
```

Именование системного вызова как `execve` — это наше высокоуровневое представление. На самом деле внутри ядра Linux, в зависимости от архитектуры процессора, этот системный вызов может называться по-разному. Например, на *x86_64* он может называться:
```bash
__x64_sys_execve
```

а на *ARM64*:
```bash
__arm64_sys_execve
```

Чтобы не привязываться к конкретной архитектуре, в *BCC* обычно указывают не полное имя функции, а только часть `execve`. Тогда *BCC* самостоятельно найдёт подходящий системный вызов для текущей архитектуры Linux. Это позволяет одной и той же eBPF-программе работать на разных системах без изменения кода:
```bash
syscall = b.get_syscall_fnname("execve")
```

Допустим, в переменную `syscall` было записано значение:
```bash
"__x64_sys_execve"
```

То есть переменная `syscall` теперь содержит реальное имя *kernel-функции*, которая вызывается при запуске новых программ:
```bash
sycall = "__x64_sys_execve"
```

Далее выполняется строка:
```bash
b.attach_kprobe(event=syscall, fn_name="hello")
```

Здесь мы говорим библиотеке BCC:  «Прикрепи мою eBPF-функцию `hello()` к *kernel-функции* `__x64_sys_execve`».

Параметр:
```bash
event=syscall
```

указывает, к какому событию нужно подключиться.

А параметр:
```bash
fn_name="hello"
```

указывает, какую именно eBPF-функцию нужно запускать при возникновении этого события.

![[Pasted image 20260521190016.png#center | 900]]

Сам механизм перехвата функции внутри ядра Linux выполняет `kprobe`. Он позволяет «подцепиться» к *kernel-функции* и автоматически вызывать нашу eBPF-программу каждый раз, когда эта функция выполняется.

К этому моменту наша eBPF-программа уже загружена внутрь ядра Linux и ожидает возникновения события `execve`. Как только какой-либо процесс вызовет системный вызов `execve()` для запуска новой программы, ядро автоматически выполнит нашу eBPF-программу.

Попробуем запустить eBPF-программу в первом терминале, а во втором выполнить обычную команду `ls`. Пока во втором терминале ничего не происходит, на первом терминале вывод пустой:
```bash
root@abigail:~/eBPF/chapter2# sudo python3 hello.py
```

Но как только во втором терминале выполняется команда `ls`, процесс `bash` вызывает системный вызов `execve()` для запуска новой программы, и attached `kprobe` сразу активирует нашу eBPF-программу:
```bash
root@abigail:~/eBPF/chapter2# sudo python3 hello.py
b'            bash-188967  [000] ...21 807140.211617: bpf_trace_printk: Hello World!'
```

То есть запуск обычной команды `ls` внутри системы автоматически вызвал выполнение нашей eBPF-программы внутри ядра Linux.

Но возникает вопрос: как мы вообще получаем результат выполнения eBPF-программы прямо в терминале? Ведь сама eBPF-программа не умеет писать в stdout, как обычные *user space*-приложения.

Вместо этого функция `bpf_trace_printk()` записывает сообщение в специальный *tracing buffer* ядра Linux. Обычно он доступен через псевдофайл:
```bash
/sys/kernel/debug/tracing/trace_pipe
```

Это не обычный файл на диске, а специальный интерфейс ядра Linux для чтения *tracing*-событий в реальном времени.

Когда в Python-коде вызывается:
```python
b.trace_print()
```

библиотека *BCC* просто начинает читать данные из `trace_pipe`, как обычный процесс читает файл, и выводит все новые сообщения в терминал. То есть ядро пишет события в *tracing buffer*, а Python-программа из *user space* читает их и показывает нам результат выполнения eBPF-программы.

### BPF Maps

Функцию `bpf_trace_printk()` обычно используют только для отладки eBPF-программ. Она очень проста в использовании: eBPF-программа записывает текстовое сообщение в специальный *tracing buffer* ядра Linux, а пользовательская программа затем читает эти сообщения и выводит их на экран. Однако такой способ подходит только для обучения и отладки, поскольку он создаёт заметный *overhead* и плохо масштабируется.

Представим, что на сервере постоянно запускаются сотни или тысячи процессов, а eBPF-программа срабатывает при каждом вызове `execve()`. В этом случае `bpf_trace_printk()` будет непрерывно записывать сообщения в *tracing buffer*:
```bash
request
request
request
request
request
...
```

При большом количестве событий объём текстовых данных быстро становится огромным. Запись и чтение такого потока создают дополнительную нагрузку на систему, а анализировать эти данные становится неудобно.

Поэтому в реальных eBPF-программах обычно используют BPF Maps. BPF Map — это специальная структура данных, расположенная в оперативной памяти ядра Linux. В отличие от `bpf_trace_printk()`, которая просто записывает текст в *tracing buffer*, BPF Maps позволяют хранить данные в структурированном виде.

Например, eBPF-программа может сохранять в Map:  
-  счётчики событий;  
-  PID процессов;  
-  сетевую статистику;  
-  значения *latency*;  
-  информацию о системных вызовах.

После этого приложение из *user space* может обратиться к этой Map через системный вызов `bpf()` и получить нужные данные напрямую из памяти ядра.

По сути, если `bpf_trace_printk()` — это аналог бесконечного вывода логов в файл, то BPF Maps — это полноценная таблица данных, через которую eBPF-программа и *user space-приложение* обмениваются информацией эффективно и с минимальными накладными расходами.

Maps можно использовать для обмена данными между несколькими eBPF-программами или для взаимодействия между приложением из _user space_ и eBPF-программой, работающей в ядре Linux. По сути, существует три основных сценария использования BPF Maps.

#### User space → eBPF-программа

Например, Python-программа из _user space_ хочет сообщить уже работающей eBPF-программе, что теперь ей нужно следить только за определённым процессом, например за `PID 1234`.

Программа в _user space_ записывает в Map:
```bash
target_pid = 1234
```

А eBPF-программа затем читает это значение:
```c
target_pid = map["target_pid"];
```

То есть в этом сценарии Map используется как механизм передачи настроек или конфигурации уже работающей eBPF-программе.

#### eBPF-программа → eBPF-программа

Допустим, мы хотим измерить время выполнения системного вызова `execve()`.

Для этого одну eBPF-программу подключаем к моменту входа в `execve()`. Её задача — сохранить текущее время в Map:
```bash
PID 123 → 1000 ns
```

Через некоторое время срабатывает другая eBPF-программа, подключённая к моменту выхода из `execve()`. Она читает сохранённое значение:
```bash
PID 123 → 1000 ns
```

Получает текущее время:
```bash
5000 ns
```

И вычисляет разницу:
```bash
5000 - 1000 = 4000 ns
```

Таким образом одна eBPF-программа сохраняет данные в Map, а другая затем использует их для дальнейших вычислений.

#### eBPF-программа → User space

Это самый распространённый сценарий использования Maps.

Например, eBPF-программа подсчитывает количество запусков процессов и сохраняет статистику в Map:
```bash
PID      COUNT
123      500
456      120
```

После этого приложение из _user space_ читает данные из Map и отображает их пользователю или передаёт дальше в системы мониторинга:
```bash
nginx: 500
postgres: 120
```

Например, эти данные могут быть показаны в терминале, отправлены в Prometheus или визуализированы в Grafana.
#### Hash Maps

Создаваемые объекты BPF Maps в памяти ядра Linux бывают разных типов. Для большинства практических задач достаточно понимать три основных типа Maps.

Hash Map — самый популярный тип BPF Maps. По своей структуре он похож на словарь `dict` в Python и хранит данные в формате «ключ → значение» (_key → value_).

Например:
```bash
PID 123 → nginx
PID 456 → postgres
```

или:
```bash
PID 123 → 500 запусков
```

В таких Maps ключом может быть PID процесса, IP-адрес, номер порта или любой другой объект, а значением — связанная с ним информация.

#### Array

Array Map представляет собой обычный массив фиксированного размера. В отличие от Hash Map, здесь ключом всегда является целое число, которое используется как индекс массива.

Например:
```bash
[0] → 100
[1] → 200
[2] → 300
```

Такой тип Maps часто используется для хранения счётчиков, статистики и других данных, к которым нужен быстрый доступ по индексу.

#### Ring Buffer

Ring Buffer отличается от предыдущих типов тем, что работает не как таблица данных, а как очередь событий. По своей идее он напоминает *tracing buffer*, о котором мы говорили ранее, однако вместо обычного текста здесь передаются структурированные данные.

Например, eBPF-программа может записать в Ring Buffer событие:
```c
struct event {
    pid = 123;
    comm = "nginx";
};
```

После этого программа из _user space_ получает событие и обрабатывает его.

Такой подход широко используется для передачи:  
-  сетевых событий;  
-  системных вызовов;  
-  событий безопасности;  
-  данных observability.

То есть если Hash Map и Array используются для хранения состояния, то Ring Buffer чаще применяется для передачи потока событий из *kernel space* в user space.

Остальными типами Maps пока можно не забивать голову. Обычно они используются для более специализированных задач:
```text
FIFO
LIFO
Bloom Filter
SockMap
DevMap
Map of Maps
Program Array
```

Для первых практических eBPF-программ понимания Hash Map, Array и Ring Buffer более чем достаточно.

#### Race Condition и Per-CPU Maps

При использовании обычных BPF Maps может возникать проблема, которая называется _race condition_ (состояние гонки).

Современный *CPU* обычно содержит несколько ядер. Например:
```text
CPU 0
CPU 1
CPU 2
CPU 3
```

Все эти ядра могут одновременно выполнять eBPF-программы. Если несколько ядер в один и тот же момент попытаются изменить одни и те же данные в общей Map, возникнет конкуренция за доступ к памяти.

Например, предположим, что в Map хранится счётчик:
```bash
counter = 100
```

*CPU 0* читает значение:
```bash
100
```

Одновременно *CPU 1* тоже читает:
```bash
100
```

Оба увеличивают счётчик на единицу и записывают результат:
```bash
101
```

В итоге одно обновление теряется, хотя ожидалось значение:
```bash
102
```

Это и есть классический пример явления *race condition*.

Для решения этой проблемы существуют Per-CPU Maps. Идея очень простая: вместо одной общей структуры данных для всех ядер Linux создаёт отдельную копию Map для каждого ядра *CPU*.

Получается примерно так:
```bash
CPU 0 → Map #0
CPU 1 → Map #1
CPU 2 → Map #2
CPU 3 → Map #3
```

Теперь каждое ядро записывает данные только в свою собственную Map и не конфликтует с другими ядрами.

Поэтому в документации часто встречаются такие пары типов:
```bash
BPF_MAP_TYPE_HASH
BPF_MAP_TYPE_PERCPU_HASH

BPF_MAP_TYPE_ARRAY
BPF_MAP_TYPE_PERCPU_ARRAY
```

Функционально они делают почти одно и то же. Разница лишь в том, что обычные Maps являются общими для всего *CPU*, а Per-CPU Maps создают отдельную копию данных для каждого ядра процессора. Это уменьшает количество блокировок, снижает вероятность *race condition* и повышает производительность на многопроцессорных системах.

Именно поэтому Per-CPU Maps очень часто используются для хранения счётчиков, статистики и метрик — там, где много ядер одновременно обновляют данные.

Иными словами мы хотим подсчтитасть какой пользователь сколько раз запускал программы и хотим записать информацию в Hash Table Map похожую на:
```bash
root  → 150
egor  → 50
john → 1000
```

#### Hash Table Map

Разберём следующую eBPF-программу:
```bash
#!/usr/bin/env python3

from bcc import BPF
from time import sleep

program = r"""
BPF_HASH(counter_table);
int hello(void *ctx) {
    u64 uid;
    u64 counter = 0;
    u64 *p;

    uid = bpf_get_current_uid_gid() & 0xFFFFFFFF;
    p = counter_table.lookup(&uid);
    if (p != 0) {
        counter = *p;
    }
    counter++;
    counter_table.update(&uid, &counter);
    return 0;
}
"""
b = BPF(text=program)
syscall = b.get_syscall_fnname("execve")
b.attach_kprobe(event=syscall, fn_name="hello")

while True:
    sleep(2)
    s = ""
    for k, v in b["counter_table"].items():
        s += f"ID {k.value}: {v.value}\t"
    print(s)
```

Этот код можно найти в разделе:
```bash
chapter2/hello-map.py
```

Как и первая программа, она подключается через `kprobe` к системному вызову `execve`. Каждый раз, когда в системе запускается новая программа, вызывается `execve()`, и наша eBPF-функция `hello()` выполняется внутри ядра.

Эта программа использует Hash Table Map. В ней ключом будет UID пользователя, а значением — счётчик количества вызовов `execve()` от этого пользователя.
```bash
BPF_HASH(counter_table);
```

Эту Map можно мысленно представить как пустую таблицу:
```bash
UID     COUNT
```

Когда пользователь запускает, например, команду `ls`, процесс вызывает `execve()`, и срабатывает eBPF-функция `hello()`. Сначала программа получает UID текущего пользователя:
```bash
uid = bpf_get_current_uid_gid() & 0xFFFFFFFF;
```

Например:
```bash
uid = 1000
```

Дальше программа ищет в Map запись с ключом `1000`:
```bash
p = counter_table.lookup(&uid);
```

Если такой ключ уже есть, `lookup()` вернёт указатель на значение счётчика в Map. Например, если в Map уже лежит:
```bash
UID     COUNT
1000    5
```

то `p` будет указывать на значение `5`.

Затем выполняется:
```bash
counter = *p;
```

Оператор `*` разыменовывает указатель `p`, то есть берёт значение из Map. В переменную `counter` записывается `5`.

Если же ключа `1000` ещё нет, `lookup()` вернёт `NULL`. Тогда условие:
```bash
if (p != 0)
```

не выполнится, и переменная `counter` останется равной `0`.

После этого программа увеличивает счётчик:
```bash
counter++;
```

Если раньше было `5`, станет `6`.  Если записи не было, станет `1`.

Далее новое значение записывается обратно в Map:
```bash
counter_table.update(&uid, &counter);
```

То есть программа говорит ядру: для ключа `UID=1000` запиши новое значение счётчика.

После нескольких запусков Map может выглядеть так:
```bash
UID     COUNT
0       150
1000    324
1001    87
```

То есть:
- root запускал программы 150 раз;  
- пользователь с UID `1000` — 324 раза;  
-  пользователь с UID `1001` — 87 раз.

Именно это и считает данная eBPF-программа: сколько раз пользователи системы запускали новые программы через `execve()`.

Теперь напишем программу на Python и запустим её в _user space_. Эта программа будет каждые две секунды обращаться к Map в ядре Linux и читать данные, которые туда записывает eBPF-программа:
```python
while True:
    sleep(2)
    s = ""
    for k, v in b["counter_table"].items():
        s += f"ID {k.value}: {v.value}\t"
    print(s)
```

Допустим, мы запускаем программу `ls`. В этом случае счётчик для нашего пользователя станет равен `1`. Если затем ещё раз выполнить `ls`, значение увеличится до `2`.

Интересный момент возникает при использовании команды `sudo ls`. В этом случае происходит сразу два вызова системного вызова `execve()`. Сначала процесс-оболочка shell запускает программу `sudo` от имени текущего пользователя с его UID, а затем уже сама программа `sudo` запускает `ls` от имени пользователя `root` с UID `0`.

Поэтому в Map появятся две записи:
```bash
1000 -> +1
0    -> +1
```

Такой подход значительно лучше варианта с использованием:
```bash
bpf_trace_printk(...)
```

поскольку eBPF-программа больше не генерирует непрерывный поток текста в _trace buffer_. Вместо этого данные сохраняются в структурированном виде внутри Map.

Однако здесь появляется другая проблема. Программа из _user space_ вынуждена постоянно опрашивать Map и спрашивать:
```text
Есть новые данные?
Есть новые данные?
Есть новые данные?
```

Такой подход называется _polling_ (опрос).

Из-за этого программа периодически обращается к ядру даже тогда, когда никаких новых событий не произошло.

Поэтому позже появились такие механизмы, как _Perf Buffer_ и _Ring Buffer_. В этом случае используется уже другая модель работы. Не приложение из _user space_ постоянно опрашивает ядро, а само ядро уведомляет приложение о появлении новых данных: "У меня появилось новое событие".

То есть вместо модели опроса используется событийная модель (_event-driven model_).

Именно поэтому большинство современных *observability*-инструментов на базе eBPF используют _Ring Buffer_ или _Perf Buffer_, а не постоянный опрос Hash Map. Такой подход создаёт меньше накладных расходов и лучше масштабируется на высоконагруженных системах.

Протестируем эту eBPF-программу. В первом терминале запускаем саму программу:
```bash
cd ~/learning-ebpf/chapter2
sudo python3 hello-map.py
```

Она начнёт раз в две секунды читать содержимое Map и выводить его в терминал.

Во втором терминале проверим свой UID:
```bash
id -u
```

У нас:
```bash
1000
```

В первом терминале появляется вывод:
```bash
ID 1000: 1
ID 1000: 1
ID 0: 1    ID 1000: 1
ID 0: 1    ID 1000: 1
```

Строка:
```bash
ID 1000: 1
```

означает, что пользователь с UID `1000` один раз запустил программу, то есть был вызван системный вызов `execve()`.

Потом появляется:
```bash
ID 0: 1    ID 1000: 1
```

Это значит, что в Map сейчас лежат две записи:
```text
UID 1000 → 1 запуск
UID 0    → 1 запуск
```

Строки повторяются потому, что Python-программа каждые две секунды заново читает текущее состояние Map и выводит его в терминал. Если новых запусков программ не происходит, содержимое Map не меняется, однако вывод всё равно будет повторяться.

В данном примере Python-программа использует метод _polling_ (опрос): она через фиксированные интервалы времени обращается к Hash Table Map и проверяет, появились ли новые данные.

Однако, как уже говорилось выше, в eBPF существуют более удобные механизмы передачи событий — _Perf Buffer_ и _Ring Buffer_. При их использовании приложение из _user space_ больше не обязано постоянно опрашивать ядро. Вместо этого ядро Linux само уведомляет программу о появлении новых данных и передаёт их ей.

В результате мы получаем не периодические снимки состояния таблицы, а поток реальных событий по мере их возникновения.

В текущем примере Python-программа просто выводит содержимое Hash Table Map. Из-за этого мы видим множество одинаковых строк, которые отражают одно и то же состояние таблицы и не позволяют легко понять, какие новые события произошли между двумя опросами.

Поэтому Hash Table Map отлично подходит для хранения состояния, счётчиков, статистики и других накопленных данных. Но если требуется наблюдать за событиями в реальном времени и видеть динамическую картину происходящего в системе, то обычно используют _Perf Buffer_ или _Ring Buffer_, поскольку они лучше подходят для передачи потока событий из ядра в _user space_ и создают меньше накладных расходов.

#### Ring Buffers

Map — это хороший инструмент для хранения состояния в памяти ядра Linux. Она отлично подходит для хранения счётчиков, статистики, конфигурации и других данных, которые нужно периодически обновлять и читать. Однако, как мы убедились выше, Map не всегда удобно использовать, когда речь идёт о непрерывном потоке событий.

В предыдущем примере Python-программа каждые две секунды опрашивала Map и проверяла, появились ли новые данные. Такой подход работает, но создаёт лишние обращения к ядру и плохо подходит для сценариев, где события возникают постоянно.

Для таких задач в eBPF обычно используют более продвинутый механизм — *Ring Buffer*.

Вместо того чтобы программа из _user space_ постоянно спрашивала ядро: «Появились ли новые данные?», гораздо эффективнее организовать работу наоборот. В этом случае eBPF-программа в _kernel space_ сама отправляет новые события в *Ring Buffer*, а программа на Python просто получает их по мере появления.

Это можно представить как очередь в магазине. Кассир выступает в роли `write pointer` и кладёт новые события в очередь. Покупатель выступает в роли `read pointer` и забирает их в том же порядке, в котором они были добавлены. Пока кассир добавляет новые события, покупатель может их последовательно читать и обрабатывать. Именно по такому принципу работает *Ring Buffer* в eBPF.


![[Pasted image 20260525100756.png#center | 900]]


В _Ring Buffer_ память организована как кольцо. Это означает, что запись данных происходит последовательно по кругу. Когда указатель записи доходит до конца выделенной области памяти, он возвращается в начало буфера. Именно поэтому этот механизм называется _Ring Buffer_ (кольцевой буфер).

Теперь логика работы eBPF-программы немного меняется. Если какой-либо процесс из _user space_ вызывает системный вызов `execve()`, в ядре возникает событие. Наша eBPF-программа подключена к этому событию через `kprobe` и получает управление каждый раз, когда такое событие происходит.

Однако теперь eBPF-программа записывает информацию не в статическую Map, а в Ring Buffer. Например, она может сохранить PID процесса, имя программы или другую информацию о событии.

В результате программа на Python больше не обязана постоянно опрашивать Map. Вместо этого она просто ожидает появления новых событий в Ring Buffer и получает их по мере поступления.

Почему это удобнее, чем Map?

Map обычно хранит текущее состояние системы:
```bash
UID 1000 → 500UID 0    → 1000
```

Из этой информации мы можем понять только итоговый результат.

Ring Buffer хранит не состояние, а последовательность событий:
```bash
PID 123 запустил nginxPID 456 запустил bashPID 789 запустил curl
```

То есть мы видим не итоговую статистику, а сам поток происходящих событий.

У Ring Buffer есть два основных указателя:
- `write pointer` — позиция, куда eBPF-программа записывает новое событие;
- `read pointer` — позиция, откуда программа из _user space_ читает события.

![[Pasted image 20260520173400.png#center | 700]]
Например, в Ring Buffer находятся три события:
```bash
[event1][event2][event3]
 ^
read
```

Python-программа читает первое событие и перемещает указатель чтения:
```bash
[event1][event2][event3]
         ^
        read
```

Затем читает второе:
```bash
[event1][event2][event3]
                  ^
                 read
```

И так далее.

Важно понимать, что *Ring Buffer* имеет ограниченный размер. Если eBPF-программа начинает генерировать события быстрее, чем программа из _user space_ успевает их читать, буфер постепенно заполняется.

Когда свободное место заканчивается, ядро больше не может записывать новые события в *Ring Buffer*, и часть событий начинает теряться.

Например:
```bash
обычно: 100 событий/сек
пиковая нагрузка: 10000 событий/сек
```

Если размер буфера рассчитан только на обычную нагрузку, то во время пиков он может переполниться. Поэтому при проектировании eBPF-инструментов размер *Ring Buffer* обычно подбирают с запасом, чтобы система могла переживать кратковременные всплески нагрузки без потери данных.

Разберем следующую eBPF-программу, код находится в:
```bash
chapter2/hello-buffer.py
```

Как и в первой eBPF-программе эта программа также будет выводить строку `Hello World`, каждый раз когда используется системный вызов `execve()`. Также эта прграмма будет получать параметры process ID и название команды которая выполняется системный вызов `execve()`:
```c
BPF_PERF_OUTPUT(output);

struct data_t {
    int pid;
    int uid;
    char command[16];
    char message[12];
};

int hello(void *ctx) {
    struct data_t data = {};
    char message[12] = "Hello World";

    data.pid = bpf_get_current_pid_tgid() >> 32;
    data.uid = bpf_get_current_uid_gid() & 0xFFFFFFFF;

    bpf_get_current_comm(&data.command, sizeof(data.command));
    bpf_probe_read_kernel(&data.message, sizeof(data.message), message);

    output.perf_submit(ctx, &data, sizeof(data));

    return 0;
}
```

Рассмотрим все этапы работа кода программы. 

Сначала создаём канал вывода:
```bash
BPF_PERF_OUTPUT(output);
```

Это создаёт специальную Map с именем `output`. Но это не *Hash Table Map* как раньше. Это *Perf Buffer Map* — канал для отправки событий из *kernel space* в *user space*. То есть логика будет следующая:
```bash
eBPF → output (Perf Buffer) → Python
```

Далее описываем формат сообщения:
```bash
struct data_t {
    int pid;
    int uid;
    char command[16];
    char message[12];
};
```

Это шаблон пакета данных.

Каждое событие будет иметь параметры:
```bash
pid
uid
command
message
```

Например:
```bash
12345
1000
ls
Hello World
```

Затем объявляется eBPF-функция:
```bash
int hello(void *ctx) {
```

Эта функция сработает каждый раз, когда будет вызван *syscall* `execve()`. `ctx` — это контекст события. Для *kprobe* он нужен, чтобы потом отправить данные через `perf_submit`.

Создаём локальные переменные:
```bash
struct data_t data = {};
char message[12] = "Hello World";
```

- `data` — пустая структура, которую сейчас заполним.
- `message` — обычная локальная строка с текстом `Hello World`.

Получаем PID:
```bash
data.pid = bpf_get_current_pid_tgid() >> 32;
```

Helper-функция возвращает 64-битное число:
```bash
старшие 32 бита → PID
младшие 32 бита → TID
```

Сдвиг `>> 32` оставляет PID. 

То есть:
```bash
data.pid = PID процесса, который вызвал execve
```

Получаем UID:
```bash
data.uid = bpf_get_current_uid_gid() & 0xFFFFFFFF;
```

Helper-функция возвращает:
```bash
старшие 32 бита → GID
младшие 32 бита → UID
```

Маска `& 0xFFFFFFFF` оставляет UID.

То есть:
```bash
data.uid = пользователь, который запустил команду
```

Получаем имя команды:
```bash
bpf_get_current_comm(&data.command, sizeof(data.command));
```

Это кладёт имя текущей команды в поле:
```bash
data.command
```

Например:
```bash
ls
bash
python3
sudo
```

Почему через `&data.command`? Потому что C-функции надо передать адрес буфера, куда записывать строку.

Копируем сообщение в структуру:
```bash
bpf_probe_read_kernel(&data.message, sizeof(data.message), message);
```

Это копирует строку `Hello World` в поле:
```bash
data.message
```

То есть теперь структура заполнена:
```bash
data.pid     = 12345
data.uid     = 1000
data.command = "ls"
data.message = "Hello World"
```

Отправляем событие в *Perf Buffer*:
```bash
output.perf_submit(ctx, &data, sizeof(data));
```

Это ключевая строка. Она говорит: отправь структуру data в output buffer. То есть eBPF не пишет в `trace_pipe`. Она пишет событие в свой *Perf Buffer*


Завершаем eBPF-функцию:
```bash
return 0;
```

Для eBPF это нормальное успешное завершение. Теперь Python-часть.

Компилируем и загружаем eBPF-программу:
```bash
b = BPF(text=program)
```

*BCC* берёт C-код из строки `program`, компилирует его и грузит в ядро.

Находим настоящее имя *syscall*:
```bash
syscall = b.get_syscall_fnname("execve")
```

Например получаем:
```bash
__x64_sys_execve
```

Цепляем eBPF-программу  к `execve`:
```bash
b.attach_kprobe(event=syscall, fn_name="hello")
```

Теперь каждый запуск программы вызывает eBPF-функцию `hello`.

Описываем обработчик события в Python:
```bash
def print_event(cpu, data, size):
```

Это callback. То есть функция, которую Python вызовет, когда из Perf Buffer придёт новое событие.

Превращаем сырые байты обратно в структуру:
```bash
data = b["output"].event(data)
```

Из Perf Buffer пришёл просто кусок байтов. BCC знает структуру `data_t`, поэтому превращает байты обратно в объект:
```bash
data.pid
data.uid
data.command
data.message
```

Печатаем событие:
```bash
print(f"{data.pid} {data.uid} {data.command.decode()} " +
      f"{data.message.decode()}")
```

Например:
```bash
12345 1000 ls Hello World
```

`.decode()` нужен, потому что `command` и `message` пришли как байты, а Python печатает строки.

Открываем Perf Buffer:
```bash
b["output"].open_perf_buffer(print_event)
```

Это значит: слушай output buffer и когда придёт событие, вызывай print_event

Бесконечно ждём события:
```bash
while True:
    b.perf_buffer_poll()
```

Python сидит и ждёт события из ядра.

Когда вы во втором терминале запускаете:
```bash
ls
```

цепочка такая:
```bash
ls
 ↓
execve
 ↓
kprobe
 ↓
eBPF hello()
 ↓
data = {pid, uid, command, message}
 ↓
output.perf_submit()
 ↓
Python perf_buffer_poll()
 ↓
print_event()
 ↓
экран
```

Главное отличие от прошлого примера: Hash Map: eBPF пишет состояние, Python сам опрашивает таблицу.

Perf Buffer: eBPF отправляет событие, Python получает его как callback. То есть этот пример уже ближе к нормальному production-паттерну: событие произошло — userspace сразу получил структурированный пакет данных.

Нам снова потребуется открыть второй терминал и выполнить любую команду, чтобы запустить всю цепочку событий. Однако обратите внимание на важное отличие от первого примера. Теперь eBPF-программа больше не записывает результаты своей работы в общий tracing-буфер ядра `trace_pipe`.

Вместо этого она отправляет события в специальный _Perf Buffer_. Фактически у нашей программы появился собственный канал передачи данных с именем `output`, через который eBPF-программа передаёт структурированные события напрямую в Python-приложение. Благодаря этому данные больше не смешиваются с другими сообщениями в `trace_pipe`, а поступают в user space через выделенный механизм доставки событий.

Вы можете убедиться, что информация не попадает в `trace pipe`, выполнив:


У нас пусто:
```bash
root@abigail:~# cat /sys/kernel/debug/tracing/trace_pipe
---
```

Чем эта eBPF-программа полезна? Используя условный `strace`, мы тоже можем перехватывать и наблюдать системные вызовы и события. Но в таком случае мы просто видим список системных вызовов. Мы не знаем: кто именно его вызвал? Какой процесс? Какой пользователь? Какая команда?

То есть теперь мы начинаем смотреть не просто на само событие, а на его контекст.

Используя eBPF, мы можем прямо внутри ядра спросить:
```bash
bpf_get_current_uid_gid()
```

и получить:
```bash
UID=1000
```

или:
```bash
bpf_get_current_comm()
```

и получить:
```bash
COMMAND=nginx
```

Именно поэтому eBPF является мощнейшим инструментом в области _observability_. Например, у нас произошла сетевая ошибка. Сама по себе эта информация мало что даёт. А вот с помощью eBPF мы можем получить полную картину:
```bash
Ошибка сети

PID=555
COMMAND=postgres
UID=26
CONTAINER=db-prod
```

Теперь мы уже понимаем, какой процесс породил проблему, под каким пользователем он работал и в каком контейнере находился.

Как вы видите, eBPF — это действительно мощнейший инструмент, но пока что наши программы только наблюдают за системой: собирают данные, записывают их в Map или *Ring Buffer*, а затем отправляют в _user space_.

Но дальше мы рассмотрим программы, которые смогут не только наблюдать за событиями, но и влиять на них прямо во время выполнения внутри ядра Linux.

Например, процесс пытается открыть файл. eBPF-программа получает информацию об этом событии, проверяет заданные правила и затем принимает решение: разрешить выполнение операции или запретить её.

Помимо демонстрации использования Ring Buffer Map, этот пример также показывает несколько eBPF helper-функций для получения контекстной информации о событии, которое запустило eBPF-программу. Благодаря этим функциям мы можем получать PID процесса, UID пользователя, имя команды и другие параметры, связанные с текущим событием.


### Упражнения

##### 1. Кто тормозит диск?

Представим, что приложение начало тормозить. Вы подключаетесь к серверу и первым делом запускаете:
```bash
top
```

Получаете следующую картину:
```bash
top - 14:23:18 up 45 days,  3:12,  2 users,  load average: 18.72, 17.95, 16.81

Tasks: 287 total,   1 running, 286 sleeping
%Cpu(s): 12.4 us,  3.1 sy,  0.0 ni, 84.0 id,  0.2 wa

PID USER      PR  NI    VIRT    RES    SHR S  %CPU %MEM     TIME+ COMMAND
1456 postgres  20   0  12.3g   4.2g  12000 S   8.0 13.4   245:12 postgres
987 nginx     20   0  85000  12000   4000 S   2.0  0.0    35:44 nginx
```

На первый взгляд всё выглядит нормально. Процессор почти не загружен, однако значение Load Average очень высокое:
```bash
Load Average = 18.72
```

Далее идём по классическому маршруту диагностики и проверяем память:
```bash
free -h
```

Получаем:
```bash
               total        used        free      shared  buff/cache   available
Mem:            32Gi        16Gi        10Gi       512Mi        6Gi        15Gi
Swap:          4.0Gi          0B       4.0Gi
```

Из этого вывода можно сделать несколько выводов:
- память не закончилась;
- Swap не используется;
- признаков OOM нет.

Значит проблема, скорее всего, не связана с памятью.

Следующий шаг — проверить дисковую подсистему:
```bash
iostat -xz 1 3
```

Получаем:
```bash
avg-cpu:  %user %system %iowait %idle
          12.3    3.1    28.7   55.9

Device            await   %util
nvme0n1           485ms   99.8%

avg-cpu:  %user %system %iowait %idle
          11.8    2.9    31.4   53.9

Device            await   %util
nvme0n1           512ms   100%

avg-cpu:  %user %system %iowait %idle
          13.0    3.2    29.9   53.9

Device            await   %util
nvme0n1           498ms   99.9%
```

Теперь картина становится гораздо интереснее.

CPU не выглядит перегруженным:
```bash
idle ≈ 55%
```

Но одновременно мы видим:
```bash
iowait ≈ 30%
```

Это означает, что процессор значительную часть времени ждёт завершения операций ввода-вывода.

Кроме того:
```bash
%util ≈ 100%
```

То есть диск практически постоянно занят.

Самое подозрительное значение:
```bash
await ≈ 500 ms
```

Для современного SSD или NVMe обычно ожидаются значения порядка:
```bash
1-5 ms
```

Здесь же каждая операция ожидает завершения почти полсекунды.

На этом этапе вывод SRE обычно выглядит так:
```bash
Проблема почти наверняка связана с диском.
```

Но мы всё ещё не знаем, какой именно процесс создаёт эту нагрузку.

Это может быть:
```bash
postgres
nginx
java
python
backup-agent
```

Обычные системные утилиты уже почти исчерпали свои возможности.

Именно здесь появляется задача для eBPF:
```text
Найти процесс и конкретный системный вызов,
которые создают задержку.
```
